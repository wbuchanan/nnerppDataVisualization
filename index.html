What is the purpose of data visualization?

  • Explore
  • Expose
  • Explain
  1. Think of explore like exploratory data analysis or the getting to know you phase of life with your data
  2. Think of expose along the lines of uncovering something new about the same data and/or artistic uses of data (e.g., data sculpting)
  3. Explain is the part that is probably most important to all of us on the call today and is probably what we all want to know how to do most effectively.

What is communication?

This diagram is based on a fairly old model of communications developed by Shannon, C. E. (1948). A mathematical theory of communication. The Bell System Technical Journal, 27, pp. 379-423. In Shannon’s case, this was a very literal translation of telecommunications, but still provides a useful framework for thinking about the process of communication (at least enough to the point where the paper has been cited > 80k times based on Google’s count).

Path A is the message as we (the transmitter) perceive/understand it.

Path B shows our effort to send this message to some recipients.

Path C is the message that the target of our communications receives.

Path D is the message as the target of the communications understands/perceives it.

In our particular case, purpose (e.g., communicating with stakeholders, communicating with C-Suite, communicating w/other data geeks, etc…) will mediate the process. However, we can use this framework to help us think through decisions along the way.

Mapping between visual and measurement scales should be one and the same

  • Anyone have any ideas about what I might mean by this statement?
  • Ask whether everyone is familiar with measurement scales.
  • Why should you care about alignment of measurement/visual scales?
  • Mention that visual scales and measurement scales sometimes have different names.
  • You're going to see what will seem to be unconventional practices, but that is because a lot of these principles get thrown to the wayside for the sake of comfort and familiarity.

Nominal Scale

  • Used for categories that have no natural order.
  • Can only test equality and inequality.
  • Ask if anyone can think of any examples.
  • Ex: School Names, Demographic groups, Subject areas, etc...
  • Defined by three axoms
  • Either a == b or a != b (testing equality/inequality)
  • If a == b then b == a (defines set equivalence in two categories)
  • If a == b and b == c then a == c (extends set equivalence to multiple categories)

Ordinal Scales

  • Used for categories that have a natural order.
  • Can test for equality/inequality.
  • Can test order of magnitude.
  • No other mathematical operators are defined on this scale.
  • Ask if anyone can think of any examples.
  • Ex: Strongly Disagree to Strongly Agree, Rankings, Letter Grades, etc...
  • In addition to nominal scale properties, two additional axoms are added
  • If a > b then b !>= a
  • If a > b and b > c then a > c (defines ordering of values)

Intervallic Scales

  • Used in cases where there is no true zero or where the location of zero is arbitrary.
  • Can test for equality/inequality.
  • Can test order of magnitude.
  • Mathematical operations are defined on this scale.
  • Ask if anyone can think of any examples.
  • Ex: Z scores, Weight of humans, scaled scores (derived from Rasch), etc....
  • Builds on the ordinal scale and adds three additional axoms
  • a + b == b + a (Commutative property)
  • If a == c & b == d then a + b == c + d (Property of Equality)
  • (a + b) + c == a + (b + c) (Associative Property)

Ratio Scales

  • Used when there is a true zero or when the location of zero is fixed.
  • Has all of the same prop
  • Ask if anyone can think of any examples.
  • Ex: % of budget remaining, # of sections offered, Raw Scores
  • Adds the last two axoms
  • a + 0 == a (defines true zero value as the absence of the quantity)
  • If a == c and b > 0 then a + b > c (defines the location of 0 to be fixed)

Visual Scales

Visual Scale type 1 2 3 4 5
Qualitative 5 1 4 2 3
Sequential 1 2 3 4 5
Divergent -2 -1 0 1 2
  • Qualitative visual scales are the visual analogs of nominal measurement scales. The colors don't convey any information beyond identifying a group.
  • Sequential scales cover ordinal, intervallic, and ratio scales based on their construction.
  • It's easiest to think of sequential scales as being primarily ordinal in nature since it is difficult to generate a color ramp that allows users to consistently perceive equally spaced intervals.
  • Divergent scales are a special type of sequential scale used to show order of magnitude in two directions.

Takeaway

  • Be careful about the selection of colors to denote groups/categories/values.
  • Be mindful of color contrast issues for any text in your visualizations.
  • Use available tools/technologies to proof your visualizations prior to publication.
  • Some technologies used to test for accessibility have known issues (e.g., computing color contrast ratios, etc...).
  • Be aware of known issues with accessibility testing software to find alternatives or test manually.

Avoid obfuscating the message

  • What types of comparisons are easy for humans to perceive?
  • What element do you want/need end users to focus on?
  • How does your choice of graph/chart support/hinder that comparison?
  • The goal here is really to get users to think about how the message they are trying to convey should drive the selection of the type of graph they use.
  • Let participants know that there are going to be some more questions that follow to look at perceptual challenges that users face.
  • Sizes of the bubbles are : 5, 7.5, 10, 20, 40, 50 in pixels
  • How many have seen and/or made a graph like this before?
  • How easy is it to identify the amount of change for the blue or orange bars?
  • Does this effectively communicate change over time?
  • What does an end user need to see or perceive in order to interpret change over time?
  • How different is the dropout rate between 2014 and 2015?
  • Area charts are not really much different from bar charts, but they do have one major difference.
  • What is the difference between the area chart and the stacked bar graph? Answer: The Area Chart includes a visual representation of the slope.
  • While it is typically recommended to start the y-axis at 0, sometimes this makes it much more challenging to identify slopes.
  • Rescaling the Y-axis makes it a bit easier to interpret the slopes, but there is still an issue.
  • Can anyone identify any of the still existing issues with this graph? Answer: because the values are summations, values that don't change over time may not have a 0 slope due to shifts in the values of other categories.
  • If we instead use a line graph we can avoid all of the issues with the effect of summations on the slopes.
  • Using visible points, we also gain some reinforcement mechanisms related to identifying the specific magnitudes that need to be compared.

Takeaways

  • Humans + Visual Comparison of Areas = Incorrect Interpretation
  • To see change over time, you need to display the slopes that represent the change.
  • Stacked bar/area charts can be challenging to interpret due to their construction.

Bind helpful information in the UI without blocking the user's view

  • If an end user isn't familiar with your graph/chart type, how do you know if they will interpret the visualization correctly/consistently?
  • How can end users actively engage with your visualization if they need additional contextual information?
  • What can you do to support multiple stakeholder groups interacting with the same visualizations?
  • How can you help end users avoid unsupported inferences/conclusions from your visualizations?
  • Explain that the examples used to have draggable elements

  • Having end points to reference like this can help end users guage how far away their interpretation is from either of the poles.

Takeaways

  • Guides to help users interpret the visualizations can drive consistent use and interpretation.
  • Make the UI elements obvious so end users can find the information easily.
  • Make the information as proximal to the visualization w/o obscuring the view.
  • Bind external resources that are regularly sought to create additional value.

Wrap Up

Wrap Up

  • Make your visualizations accessible to all potential end users whenever possible.
  • Reinforce the underlying meaning of the data with the visual mapping.
  • Humans and visual comparison of areas don't mix.
  • Empower the end users to make appropriate inference from your visualization with supporting materials.
  • Create/implement visualization standards to provide end users with consistency.
  • Make sure to mention not using dual axes or reverse axes, etc...
